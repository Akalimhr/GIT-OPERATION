理清一个 Git 中最关键的底层逻辑：**分支和 `merge` 操作，只认 `commit`，不认 `add`。**

------

### 核心概念

1. **`commit`（提交）**：这是真正的“保存快照”。只有执行了 `commit`，你的修改才真正属于当前分支。此时你切换回 `master`，工作目录会被 `master` 的快照替换，你确实**看不到**刚才的修改。
2. **`add`（暂存）但未 `commit`**：`add` 只是把文件放到了一个“公共的等待区（暂存区）”。如果你只 `add` 了文件而没有 `commit`，当你尝试切换到 `master` 时：
   - **情况 A**：如果这个文件在两个分支里没有冲突，Git 会**把这些未提交的修改强行带到 `master` 分支上来**（你会发现在 master 上也能看到修改）。
   - **情况 B**：如果有冲突，Git 会直接**报错并拒绝你切换分支**，提示你需要先 commit 或者 stash（暂存）。

**结论**：在切换分支或进行合并之前，**一定要确保当前工作已经 `commit`（或放入 stash）**，保持工作区干净（Working tree clean）。

------

### 解答你的疑问：“平行线”下的各种 Merge 情况

你问到：如果两个平行线（分支），经历的操作步数不一样（一个 add，一个 commit，或者一个好几次 commit），`merge` 会是什么情况？

记住上面说的原则：**`merge` 只合并不同分支的 `commit` 历史**，不关心你 `add` 了什么，也不关心你提交了多少次。

#### 场景 1：分支 A 有 1 次 `commit`，主干 `master` 只有 `add` 或啥也没干

- **如果 `master` 啥也没干**：这叫 **Fast-forward（快进合并）**。因为 `master` 完全没动过，Git 直接把 `master` 的指针“快进”到分支 A 的最新位置，不会产生任何冲突，完美合并。
- **如果 `master` 有未提交的修改（只 `add` 了或修改了文件）**：
  - Git 会尝试将分支 A 的 `commit` 内容合并到你的工作目录中。
  - 如果 A 修改的文件和你正在修改的文件**不是同一个**，合并成功，你的未提交修改保留。
  - 如果 A 修改的文件和你正在修改的文件**是同一个**，Git 为了保护你还没 `commit` 的心血，会**直接拒绝 merge**（报错：`Your local changes to the following files would be overwritten by merge`）。

#### 场景 2：分支 A 经历了 2 次（甚至 100 次）`commit`，分支 B 经历了 1 次 `commit`

- **Git 不在乎你 commit 了多少次**。当你执行 `git merge B` 时，Git 会自动找到 A 和 B 最近的一个“共同祖先”（也就是它们分叉时的那个状态）。
- 然后，Git 会对比 A 的**最终快照**和 B 的**最终快照**：
  - 如果改了不同的地方 -> 自动合并，生成一个新的“合并提交”（Merge Commit）。
  - 如果改了同一个地方 -> 提示代码冲突（Conflict）。此时工作区的文件会变成包含 `<<<<<<<` 和 `>>>>>>>` 标记的状态，需要你手动修改保留想要的代码，然后执行 `git add` 标记解决，最后 `git commit` 完成合并。

### Git 分支管理详细总结

基于你的笔记和上面的澄清，我们可以将 Git 分支管理总结为以下几个核心阶段：

#### 1. 核心理念：快照与指针

- Git 并不为每个分支复制一份完整的目录，分支本质上只是一个指向某个 `commit`（快照）的**轻量级指针**。
- `HEAD` 是一个特殊的指针，它指向你当前正在工作的本地分支。切换分支（`git checkout` 或 `git switch`），本质上就是移动 `HEAD` 指针，并用该指针指向的快照**替换你的工作目录**。

#### 2. 日常工作流（隔离开发）

- **创建并切换**：使用分支可以将新功能开发、Bug 修复与主枝干（`master`/`main`）切分开来，互不干扰。
- **保存状态**：在分支上工作时，必须完成 `git add` -> `git commit` 的完整闭环，修改才会被真正记录在该分支的快照中。只有这样，切换分支时才不会发生文件“串门”或切换失败的情况。

#### 3. 合并工作流（汇聚成果）

当你切回主干（`git checkout master`）并希望将分支成果合入时，执行 `git merge <分支名>`。合并分为两种情况：

- **快进合并 (Fast-forward)**：你切出分支后，`master` 没有任何新的 `commit`。Git 直接把 `master` 指向你分支的最新位置。
- **三方合并 (3-Way Merge)**：你在分支开发的同时，`master` 上也有了新的 `commit`（两人平行开发）。Git 会将两个分支的最新状态和它们的共同祖先进行比对，自动合并非冲突内容，并生成一个新的 `commit`。

#### 4. 冲突解决机制

如果在“三方合并”时，两个分支修改了同一个文件的同一部分，就会产生**冲突 (Merge Conflict)**：

1. Git 会暂停合并进程，并在有冲突的文件中插入冲突标记（`<<<<<<<` 等）。
2. 你需要打开文件，人工决定保留哪些代码，删除这些标记。
3. 修改完成后，使用 **`git add <文件名>`** 告诉 Git：“我已经解决了这个文件的冲突”。
4. 最后执行 **`git commit`**，彻底完成这次冲突解决和合并过程。

#### 5. 善后工作

- 合并成功后，原分支的历史已经包含在主干中，原分支的使命结束。
- 使用 `git branch -d <分支名>` 可以安全删除该分支。如果有未合并的内容，Git 会阻止你删除；若要强删，需使用 `-D` 参数。
