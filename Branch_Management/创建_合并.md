### Git 分支管理总结test

#### 一、核心概念

- **分支本质**：Git 将每次提交串成时间线，即分支。默认主分支为 `master`。
- **HEAD 指向**：HEAD 不直接指向提交，而是指向当前分支（如 `master`），分支指针再指向具体提交。
- **操作效率**：创建 / 切换分支仅改变指针指向，工作区文件不变，速度极快。

#### 二、基础分支操作（dev 分支示例）

1. **创建并切换分支**
   - 命令：`git checkout -b dev`（等价于 `git branch dev` + `git checkout dev`）
   - 查看分支：`git branch`（当前分支前标 `*`）
2. **在 dev 分支提交**
   - 修改文件 → `git add <file>` → `git commit -m "说明"`
3. **切换回 master**
   - 命令：`git checkout master`（此时 dev 的修改在 master 不可见）
4. **合并 dev 到 master（快进模式）**
   - 命令：`git merge dev`（显示 `Fast-forward`，直接移动 master 指针到 dev 提交）
5. **删除 dev 分支**
   - 命令：`git branch -d dev`

#### 三、冲突合并操作（feature1 分支示例）

1. **创建并切换 feature1 分支**

   - 命令：`git switch -c feature1`（现代 Git 推荐，与 `checkout -b` 等价）

2. **在 feature1 分支提交**

   - 修改同一文件（如 readme.txt）→ `git add` → `git commit`

3. **在 master 分支独立提交**

   - 切换回 master：`git switch master`
   - 修改同一文件的同一位置 → `git add` → `git commit`（此时两分支分叉，无法快进合并）

4. **合并 feature1 到 master（解决冲突）**

   - 命令：`git merge feature1` → 提示 `CONFLICT`（内容冲突）
   - 查看冲突：文件中用 `<<<<<<< HEAD`（当前分支）、`=======`（分隔）、`>>>>>>> feature1`（待合并分支）标记冲突
   - 解决冲突：手动修改文件 → 保存 → `git add <file>` → `git commit -m "conflict fixed"`

5. **查看合并历史**

   

   - 命令：`git log --graph --pretty=oneline --abbrev-commit`（图形化显示分支合并）

   

6. **删除 feature1 分支**

   - 命令：`git branch -d feature1`

#### 关键总结

- 分支操作高效：仅操作指针，不改变工作区文件。
- 快进合并：分支未分叉时直接移动指针。
- 冲突解决：手动修改冲突文件后提交，完成合并。
